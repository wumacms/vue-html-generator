# 静态站点生成模块 — 实现思路（结合建站平台实际需求）

本文档在平台设计（多用户、网站/页面/区块、JSON 配置化）和静态生成技术选型基础上，**进一步细化「静态站点生成模块」的实现思路**：模块边界、数据流、管线步骤、目录结构、与平台/发布的衔接。

> 前置文档：[平台设计思路-多用户与JSON配置化.md](./平台设计思路-多用户与JSON配置化.md)、[可视化建站-静态生成与发布分析.md](./可视化建站-静态生成与发布分析.md)。

---

## 一、模块定位与边界

### 1.1 模块是什么

**静态站点生成模块**：在**构建时**接收「某个网站的完整配置数据（网站 JSON）」，输出**可直接部署到 GitHub Pages 的静态文件目录**（多页 HTML + 可选 CSS/JS/图片等）。

- **输入**：一份「网站 JSON」（含站点元信息、全部页面、每页的完整区块 JSON）。
- **输出**：`dist/` 目录，例如 `dist/index.html`、`dist/about/index.html`、`dist/assets/`。
- **不负责**：用户登录、网站 CRUD、数据库、API 鉴权、GitHub OAuth、最终推送到 gh-pages（可由上游「发布服务」或 CI 负责）。

### 1.2 在整体链路中的位置

```
建站平台前端/后端
       │
       │ 用户点击「发布」→ 后端拉取完整网站 JSON（或前端导出）
       ▼
┌──────────────────────────────────────┐
│  静态站点生成模块（本模块）            │
│  输入：网站 JSON                       │
│  处理：逐页 Vue renderToString → HTML │
│  输出：dist/                          │
└──────────────────────────────────────┘
       │
       │ 上游或 CI 将 dist/ 推送到 GitHub（gh-pages / Actions）
       ▼
  GitHub Pages 对外服务
```

模块只做「JSON → 静态文件」；谁在何时调用、以及发布到哪，由平台或 CI 决定。

## 二、输入：如何拿到「网站 JSON」

### 2.1 要求形态

生成模块要求拿到的是一份**可直接渲染的网站 JSON**：

- 包含该站点的 `name`、`basePath`、`theme`、`seo` 等；
- `pages` 为**数组**，每项为**完整页面 JSON**（含 `path`、`title`、`meta`、`layout` 等）；
- 每个页面中的 `blocks` 为**完整区块 JSON 数组**（含 `type`、`props`、`styles`、`order` 等），**不能只是区块 id 列表**（否则模块内还要调 API，耦合过重）。

即：**模块不查库、不调业务 API**；上游在调用生成前，负责把「页面/区块 id」解析为「完整页面/区块 JSON」并组装成一份网站 JSON。

### 2.2 输入来源方式（由上游实现，模块只约定接口）

| 方式 | 说明 | 模块侧 |
|------|------|--------|
| **A. 本地文件** | 平台「导出网站」生成 `site.json`，生成脚本读该文件。 | 支持从路径读取 JSON，或 stdin。 |
| **B. API 一次拉取** | 后端提供 `GET /api/sites/:id/export` 返回完整网站 JSON（已组装好 pages/blocks）。 | 支持从 URL 拉取（或由上游 curl 下载后以文件形式传入）。 |
| **C. 环境变量/参数** | 构建时传入 `siteId`，脚本内部调平台 API 拉取（不推荐，模块与 API 强耦合）。 | 可选支持；更推荐 A/B，由上游组装好后传入。 |

**推荐**：模块**只接受「完整网站 JSON」**（文件路径或已读入的对象）；由**后端或前端**在「发布」流程里负责调用平台 API、组装完整网站 JSON，再写入文件或传给生成脚本。

### 2.3 网站 JSON 结构约定（与平台设计对齐）

生成模块与平台预览共用同一套结构，例如：

```ts
// 类型示意，便于实现时校验
interface SiteJSON {
  id: string;
  name: string;
  basePath?: string;   // 如 '/repo-name/'，用于 GitHub Pages 子路径
  theme?: string;
  seo?: { title?: string; description?: string };
  pages: PageJSON[];
}

interface PageJSON {
  id: string;
  path: string;        // 如 '/', '/about'，决定输出路径
  title: string;
  meta?: { description?: string };
  layout?: string;
  blocks: BlockJSON[]; // 已展开的完整区块，按 order 排序
}

interface BlockJSON {
  id: string;
  type: string;        // 对应内置区块模板：hero, features, cta, footer...
  props: Record<string, unknown>;
  styles?: Record<string, string>;
  order: number;
}
```

- `path` 需满足：以 `/` 开头、无尾斜杠（或统一约定）；生成时据此得到 `dist/index.html`、`dist/about/index.html` 等。
- `blocks` 在传入前应按 `order` 排好序，模块内可不再排序或做二次排序（按需）。

## 三、输出：目录与文件形态

### 3.1 目录结构

```
dist/
├── index.html              # path === '/' 的页面
├── about/
│   └── index.html          # path === '/about'
├── blog/
│   └── post-1/
│       └── index.html      # path === '/blog/post-1'（若采用此类 path）
├── assets/                 # 可选：共享资源
│   ├── style.[hash].css
│   └── runtime.[hash].js   # 若需交互再引入
└── images/                 # 可选：本地化后的图片等
```

- 每个 `path` 对应一个 HTML 文件；规则：`path === '/'` → `dist/index.html`，`path === '/about'` → `dist/about/index.html`，以此类推（path 转成相对目录 + `index.html`）。
- 所有静态资源引用（CSS/JS/图片）需考虑 `basePath`（见下）。

### 3.2 basePath 与资源引用

- 若网站配置了 `basePath`（如部署在 `https://user.github.io/repo-name/`），则：
  - 在 HTML 模板中设 `<base href="${basePath}">`；
  - 所有 `<link href="...">`、`<script src="...">`、`<img src="...">` 使用相对路径或「basePath + 相对路径」，确保发布后资源可访问。
- 模块在渲染时可通过 provide/inject 或全局配置把 `basePath` 传给区块组件，区块内图片等 URL 统一加前缀。

### 3.3 单页 HTML 模板要素

每页生成的 HTML 建议包含：

- `<!DOCTYPE html>`、`<html>`、`<head>`、`<body>`；
- `<meta charset="utf-8">`、视口、当前页 `title` / `meta.description`（来自页面 JSON 或站点 SEO）；
- `<base href="...">`（若有 basePath）；
- 占位符替换为当前页的 **app HTML**（即 PageRenderer 的 renderToString 结果）；
- 可选：全局样式 `<link href="assets/style.xxx.css">`；
- 可选：需要交互的页面再加 `<script src="assets/runtime.xxx.js">`。

## 四、核心管线（实现步骤）

### 4.1 总流程

1. **加载与校验网站 JSON**  
   从文件或参数读入，校验必填字段（如 `pages` 为非空数组、每页含 `path` 和 `blocks`）。无效则直接报错退出。

2. **规范化数据**  
   - 对每页 `blocks` 按 `order` 排序（若上游未排）；  
   - 将 `path` 转为输出相对路径（path → 文件路径映射，见下）；  
   - 可选：校验每个 block 的 `type` 是否在已注册的区块模板集合内，未知 type 可降级为占位或报错。

3. **准备渲染环境（一次）**  
   - 创建 Vue 应用工厂（或单例应用，每页只换 props）：`createSSRApp(PageRenderer, { page, site })`；  
   - 注册所有内置区块组件（BlockRenderer 内部按 `type` 映射到具体组件）；  
   - 若使用 Vite，在 Node 中通过 `vite.createSSRContext` 或 `createServer` 的 `ssrLoadModule` 加载入口，再执行 `renderToString`（具体见 Vite SSR 文档）。

4. **逐页渲染与写入**  
   - for 每页 `page` in `site.pages`：  
     - 设置当前页的 props（如 `page`, `site`, `basePath`）；  
     - `renderToString(app)` 得到 `appHtml`；  
     - 将 `appHtml` 填入 HTML 模板，并写入 `dist/${pathToFile(page.path)}/index.html`（或根目录 `index.html`）。  
   - 若有全局 CSS/JS，在模板中引用同一套 assets 路径（带 basePath）。

5. **资源与收尾**  
   - 若使用 Vite 打包：先 build 出 `assets/style.css`、`assets/runtime.js`（若需要），再运行上述 Node 脚本；或脚本内通过 Vite 的 SSR 构建模式产出资源。  
   - 图片：区块 `props` 中多为图片 URL。若保持外链，则无需写入 `dist/images/`；若需「本地化」则额外步骤：下载并写入 `dist/images/`，替换 props 或输出 HTML 中的 URL 为相对路径（可选，按产品需求做）。

### 4.2 path → 文件路径映射

建议规则：

- `path === '/'` → `dist/index.html`；
- `path === '/about'` → `dist/about/index.html`；
- `path === '/blog/hello'` → `dist/blog/hello/index.html`。

即：去掉首尾斜杠后，若为空则 `index.html`，否则 `${pathSegment}/index.html`。注意过滤非法 path（如 `..`、空段等），避免路径穿越。

### 4.3 与平台组件复用

- **BlockRenderer**：根据 `block.type` 映射到「内置区块模板」Vue 组件（如 `type: 'hero'` → `HeroBlock.vue`）。  
- **PageRenderer**：接收 `page`（页面 JSON）、`site`（网站 JSON），内部 `v-for` 渲染 `blocks`，每个 block 用 `BlockRenderer`。  
- 这些组件应与**建站平台预览**共用同一套代码（同一仓库下的包、或 npm 私有包），保证「预览即发布」一致。  
- 生成模块的 Node 脚本通过 Vite SSR 或直接 `import` 这些组件，在 Node 环境中只做 `renderToString`，不依赖 DOM/Browser API；若区块组件内有 `onMounted`/浏览器 API，需在 SSR 时做兼容（如仅客户端执行或用 mock）。

## 五、技术实现要点（Vite + Vue 3）

### 5.1 渲染入口

- 入口组件：例如 `App.vue` 或 `PageRenderer.vue`，接收 `page`、`site` 为 props。  
- 在 Node 中：  
  - `createSSRApp(PageRenderer, { page, site })`；  
  - 注册所有区块组件为全局或通过异步组件按 type 加载；  
  - `renderToString(app)` 得到 HTML。  
- 若用 Vite：可用 `vite build --ssr` 打出 SSR 入口 bundle，再在 Node 里 `require` 该 bundle 并执行渲染；或使用 `vite.ssrLoadModule` 在开发/构建时加载源码（需配置 SSR 的 entry、external 等）。

### 5.2 HTML 模板

- 模板可放在仓库内一静态文件（如 `template.html`），占位符如 `<!--app-html-->`；  
- 每页渲染后：读模板 → replace 占位符为 `appHtml`，并填入 `title`、`meta`、`basePath`、assets 的链接；  
- 或使用简单的模板库（如 `ejs`、`minimist`）或字符串拼接，避免过重依赖。

### 5.3 样式与脚本

- **全局样式**：建站平台或主题的全局 CSS，在生成前用 Vite 打成一笔 `assets/style.[hash].css`，模板中统一引用。  
- **区块内样式**：若区块有 `styles` 对象，可在 BlockRenderer 中渲染为 `<div :style="block.styles">`，无需单独文件。  
- **脚本**：若部分区块需要交互（如表单、轮播），可打一份 runtime chunk，仅在需要该能力的页面引入（或全站引入，按需优化）。

### 5.4 错误与边界

- 未知区块 `type`：记录 warning，渲染为占位 div 或跳过，避免整站生成失败。  
- 某页 `blocks` 为空：仍输出该页 HTML（仅布局/导航等）。  
- `path` 重复：以首次或最后一次为准，并打 warning。  
- 文件写入失败：抛出错误，便于上游/CI 处理。

## 七、触发与发布衔接

### 7.1 谁在何时调用生成模块

- **方式 1（推荐）**：用户点击「发布到 GitHub Pages」→ 后端拉取完整网站 JSON，写入临时文件或对象，调用生成脚本（如 `node scripts/generate.js --input /tmp/site-xxx.json --out dist`），再将 `dist/` 打包或推送到 GitHub（如通过 GitHub API 推 gh-pages、或触发仓库的 GitHub Actions）。  
- **方式 2**：平台提供「导出网站 JSON」→ 用户下载 `site.json`，在本地或自有 CI 中运行生成脚本，再自行推送 `dist/` 到 GitHub。  
- **方式 3**：站点数据与代码同仓（如每个站点一个 repo），通过 GitHub Actions 在 push 时拉取 API 或读仓内 `site.json`，运行生成脚本并用 `actions-gh-pages` 推送。

模块本身只需支持：**输入 = 网站 JSON（文件或 stdin）、输出 = dist 目录**；不关心鉴权、Git 推送、GitHub Token。

### 7.2 与 GitHub Pages 的衔接

- 生成结果中的 `basePath` 需与最终仓库的 GitHub Pages 路径一致（如 `/<repo>/`）。  
- 部署时只需把 `dist/` 内容推到 gh-pages 分支或默认分支的 `docs/`（按 GitHub 规则）；可使用 `peaceiris/actions-gh-pages` 等 Action，或后端用 GitHub API 提交到 gh-pages。  
- 多页静态站无需 SPA fallback；确保每个页面链接、资源链接都带对 basePath 即可。

### 7.3 可视化建站平台「一键发布到 GitHub Pages」的实现思路

在可视化建站平台里，用户点击「一键发布」即可把当前站点发布到 GitHub Pages，可以沿用当前项目的「生成 + GitHub Actions 部署」思路，用以下方式之一实现。

| 方式 | 做法 | 优点 | 注意点 |
|------|------|------|--------|
| **A. 平台提交 site.json + 触发 workflow** | 用户先绑定 GitHub 仓库（OAuth 或 PAT）。点击发布时：① 平台后端拉取完整网站 JSON；② 用 GitHub API 将 `site.json` 提交到该仓库的默认分支；③ 同一仓库内已有本项目的 deploy workflow（或平台提供模板仓库），可 **再调用 `workflow_dispatch`** 或依赖 **push 触发**，由 Actions 执行 `build:ssr` → `build:css` → `generate` → 推 gh-pages。 | 生成在 GitHub CI 里跑，与当前 [deploy.yml](../.github/workflows/deploy.yml) 一致；无需平台跑 Node 生成。 | 需要用户授权仓库写权限；每次发布会在仓库里产生一次 site.json 的 commit。 |
| **B. repository_dispatch + workflow 内拉取 JSON** | 用户绑定仓库后，该仓库的 workflow 除 `push`/`workflow_dispatch` 外，再增加 `repository_dispatch`（如 `event_type: publish-site`）。用户点击发布 → 平台后端用 GitHub Token 调 `POST /repos/:owner/:repo/dispatches`，在 `client_payload` 里带 `site_id`（及可选的一次性拉取 token）；workflow 触发后先请求平台 API 用 `site_id` 拉取完整网站 JSON，写入 `site.json`，再执行生成与部署。 | 不在仓库里提交 site.json，数据始终在平台；一次点击即发布。 | workflow 需能访问平台 API（鉴权、网络）；平台需提供「按 site_id 返回完整网站 JSON」的接口（可带一次性 token）。 |
| **C. 平台后端直接生成并推 gh-pages** | 用户点击发布 → 平台后端拉取网站 JSON，在服务器上执行与当前项目相同的生成命令得到 `dist/`，再用 GitHub API（用户 PAT 或 GitHub App）把 `dist/` 内容推送到用户仓库的 gh-pages 分支。 | 不依赖用户仓库里是否有 Actions；发布流程完全在平台侧。 | 平台必须有 Node 环境并跑生成脚本；需用户授权写仓库（或 GitHub App 安装到用户仓库）。 |

**推荐组合**：  
- 若希望「生成逻辑与当前项目完全一致、且不希望在平台跑 Node」：采用 **A** 或 **B**（A 实现更简单，B 不在仓库留 site.json）。  
- 若平台本身就有 Node 服务、且希望发布不依赖用户仓库里的 workflow：采用 **C**。

无论哪种方式，生成模块的约定不变：**输入 = 完整网站 JSON，输出 = dist/**；平台或 CI 负责把「网站数据」交给生成器，并把「dist/」推到 GitHub Pages。

## 八、小结

| 项 | 说明 |
|----|------|
| **输入** | 一份完整「网站 JSON」（含 pages、每页含完整 blocks），由上游 API 或导出提供；模块不查库、不调业务 API。 |
| **输出** | `dist/`：每页一个 HTML 文件（path 映射为目录 + index.html），可选 assets（CSS/JS/图片）。 |
| **核心逻辑** | 加载并校验网站 JSON → 逐页用 Vue PageRenderer + BlockRenderer 做 renderToString → 填入 HTML 模板并写入文件；资源引用与 basePath 一致。 |
| **组件** | 与平台预览共用 BlockRenderer / PageRenderer 及内置区块模板，保证所见即所得。 |
| **技术** | Vite + Vue 3，Node 侧仅用 SSR API 做构建时渲染；不部署 SSR 服务器。 |
| **触发与发布** | 由平台后端或 CI 在「发布」时调用生成脚本；发布到 GitHub Pages 由上游或 CI 负责，模块只产出 dist/。 |

按上述思路即可把「静态站点生成」做成一个边界清晰、可被建站平台或 CI 调用的独立模块，并与现有平台设计（多用户、网站/页面/区块、JSON 配置化）对齐。

## 九、FAQ：为何将生成功能放到 API 中

**问：为什么要将生成静态站点的功能放到一个 API 中？**

**答：**

把「生成静态站点」做成 API，本质是为了**让浏览器里的页面能触发生成**，同时**把「谁在跑生成」和「谁在点按钮」分开**。

### 1. 为什么需要 API？

- **生成逻辑只能在 Node 里跑**：要读 `site.json`、用 `dist/ssr/entry-server.js` 做 Vue SSR、写 `dist/` 文件，这些都是 Node 环境才能做的事，浏览器里的 JS 做不到。
- **用户操作在浏览器里**：你希望用户在「管理页」上点「生成静态站点」按钮，而不是去终端敲命令。
- **把两者连起来**：页面在浏览器，生成在 Node，中间就要有一个「桥」—— 浏览器发请求，Node 收到后执行生成并返回结果。这个「请求 → 执行 → 返回」的桥，用 HTTP API 来实现是最直接、也最通用。

所以：**不是「故意做成 API」**，而是「要在网页上点按钮触发生成，就必须有一个服务端接口，用 HTTP 暴露成 API 是最自然的形式」。

### 2. 用 API 带来的好处

- **前后端分离**：管理页（Vue 前端）只负责 UI 和发请求；生成逻辑、文件读写都在服务端，职责清晰。
- **复用同一套生成逻辑**：CLI（`node src/cli.js`）和「页面按钮」都通过同一套 `runGenerate()`；API 只是多了一层 HTTP 包装，不重复写生成代码。
- **以后扩展方便**：如果以后有别的客户端（例如别的站点、CI、桌面工具）要触发生成，只要调同一个 `POST /api/generate` 即可，不用再为每种方式写一套调用方式。
- **错误和结果统一返回**：成功/失败、页数、错误信息都可以用 JSON 返回，前端可以统一处理（例如成功提示、错误展示）。

### 3. 和「不用 API」的对比

- **不用 API 的替代**：例如在本地用 Electron 或 Tauri 做桌面应用，在进程内直接调 `runGenerate()`，不经过 HTTP。那样也可以，但会变成「桌面应用」而不是「在现有网页里点按钮」；你当前是「一个 Vue 管理页 + 一个 Node 服务」，用 API 把两者连起来是最简单、一致的方案。
- **CLI 和 API 的关系**：CLI 适合脚本、CI、本机终端；API 适合「已经有一个 Web 服务在跑、用户在浏览器里操作」的场景。两者共用 `runGenerate()`，只是入口不同（命令行参数 vs HTTP 请求）。

### 4. 小结

把生成静态站点放到 API 里，本质原因是：**生成必须在 Node 里执行，而触发来自浏览器**，所以需要服务端暴露一个接口；用 HTTP API 是实现「浏览器触发 → Node 执行」的标准、简单、可扩展的方式，而不是为了「做成 API」而做的设计。

## 十、FAQ：生成项目能否部署到 GitHub Pages

**问：按照这种分析，当前生成静态站点的项目是不能部署到 GitHub Pages 进行使用的对吗？**

**答：**

要分两部分说：

- **能部署到 GitHub Pages 的**：是「生成结果」`dist/`（静态 HTML + CSS 等）。这就是本项目的目标产物，完全可以部署到 GitHub Pages 使用。
- **不能部署到 GitHub Pages 的**：是「管理页 + 生成 API」这一套（Vue 管理页 + `src/server.js`）。因为 GitHub Pages 只提供静态托管，不跑 Node，所以没人能处理 `POST /api/generate`，点「生成静态站点」必须有一个在跑的 Node 服务。

因此：

- **按前面的分析**：「在页面上点按钮生成」依赖「浏览器 → API → Node 执行」，所以**带按钮和 API 的这套工具**本身不能放到 GitHub Pages 上运行，只能在本机或任意有 Node 的环境里跑（例如 `pnpm run dev`）。
- **当前项目的两种用法**：
  1. **本地/服务器**：运行 `pnpm run dev` → 打开管理页 → 点按钮生成 → 得到 `dist/`。这一整段不能部署到 GitHub Pages。
  2. **把生成好的站点部署出去**：把 `dist/` 里的内容（例如用 GitHub Actions 或手动）推到 gh-pages 或 `docs/`。这部分完全可以部署到 GitHub Pages，供访客访问。

**一句话**：能部署到 GitHub Pages 的是「生成出来的静态站」`dist/`；不能部署上去的是「带生成 API 的管理页/项目本身」。所以是「生成静态站点的**产物**能部署到 GitHub Pages；**用来点按钮生成的那套项目**不能部署到 GitHub Pages」。

## 十一、FAQ：管理页和 API 服务要部署到哪里

**问：那么这个静态生成项目的管理员和 API 服务要部署到哪里呢？**

**答：**

管理页和 API 服务（带「生成静态站点」按钮的那套）不能放在 GitHub Pages，可以部署到下面几类环境。

- 适合：个人站、更新不频繁、只有你自己会生成。

### 2. 有 Node 的服务器（VPS / 云主机）

- 把整个项目（或至少 `src/server.js` + `dist-admin/` + `dist/ssr/`、`site.json` 等运行依赖）部署到一台能跑 Node 的机器（如腾讯云、阿里云、AWS EC2、DigitalOcean 等）。
- 用 `node src/server.js` 或 pm2 等常驻进程跑起来，通过 `http://你的域名:3579` 访问管理页并调 API。
- 适合：希望从任意地方打开网页点按钮生成、或多人共用一套生成环境。

### 3. 支持 Node 的 PaaS

- 部署到能跑 Node 的托管平台，例如：**Railway**、**Render**、**Fly.io**、**Heroku** 等。
- 平台负责进程和 HTTPS，你只要把项目（含 `server.js`、构建好的 `dist-admin` 等）按平台要求部署即可。
- 适合：不想自己管服务器，又希望管理页和 API 在公网可访问。

### 4. 不部署管理页，只用 CI 生成（推荐之一）

- **不把管理页/API 部署到任何地方**，只在需要发布时用 **GitHub Actions**（或其它 CI）跑生成：
  - 在仓库里放好 `site.json`（或从别处拉取）。
  - workflow 里执行 `pnpm run build:ssr && pnpm run build:css && pnpm run generate`，把得到的 `dist/` 推到 gh-pages 或 `docs/`。
- 触发方式：push 到某分支、手动 Run workflow、或通过 GitHub 的 `workflow_dispatch` 在网页上点一下「运行」即可，相当于「在网页上点一下就能生成并发布」。
- 适合：内容以仓库/配置为主、不需要常驻一个「管理后台」，且希望全部在 GitHub 内完成。

**小结**：要「把管理页和 API 部署出去」→ 选 **VPS** 或 **PaaS（Railway / Render / Fly.io 等）**。不一定要部署管理页 → 用 **本机 `pnpm run dev`** 或 **GitHub Actions 等 CI** 生成，只把 **`dist/` 部署到 GitHub Pages**。当前仓库没有写死「必须部署到哪」；按你是想「公网管理页」还是「本机/CI 生成」来选上面一种即可。

## 十二、不部署管理页、只用 CI 生成的实现方式（详细）

本节对 FAQ 十一中的「方式 4：不部署管理页，只用 CI 生成」做详细说明。

### 一、思路是什么

- **不部署**：不把管理页和 `node src/server.js` 部署到任何服务器，也不在公网提供「点按钮生成」的页面。
- **只用 CI 生成**：在 GitHub 的 CI（例如 GitHub Actions）里执行和本地一样的生成命令，得到 `dist/`，再由 CI 把 `dist/` 推到 GitHub Pages（或 `docs/`）。
- 对你来说，「发布」= 在 GitHub 上点一下运行 workflow，或 push 到某分支，就自动生成并更新站点，无需本机跑 `pnpm run dev`。

### 二、整体流程（一步步在做什么）

1. **准备输入**  
   在仓库里有一份「网站 JSON」（例如根目录的 `site.json`），或由 workflow 从别处拉取/生成。CI 就用这份数据生成静态站。

2. **触发 workflow**  
   例如：push 到 `main`、或你在 GitHub 网页上对某个 workflow 点「Run workflow」（`workflow_dispatch`）。

3. **在 CI 里跑生成**  
   - 装依赖：`pnpm install`  
   - 构建 SSR 和样式：`pnpm run build:ssr`、`pnpm run build:css`  
   - 生成静态站：`pnpm run generate`（读 `site.json`，写 `dist/`）

4. **把结果部署出去**  
   - 把当前仓库里的 `dist/` 推送到 **gh-pages 分支**（或默认分支下的 **docs/**），具体按 GitHub Pages 的配置来。  
   - 常用做法：用官方或第三方 Action（如 `peaceiris/actions-gh-pages`）把 `dist/` 当作「要发布的目录」推上去。

5. **结果**  
   GitHub Pages 用新内容更新站点，访客看到的是最新生成的静态站，全程没有管理页和 API 参与。

### 三、仓库里需要准备什么

- **网站数据**  
  - 至少有一份可用的 `site.json`（或 workflow 里能生成/下载到同结构的 JSON）。  
  - 若站点内容来自 CMS/API，可以在 workflow 里先请求 API 生成 `site.json` 再跑 `pnpm run generate`。

- **生成与部署脚本**  
  - 能正常执行：`build:ssr`、`build:css`、`generate`（即 `export` 的前几步）。  
  - 若你希望「一条命令搞定」，可以用：  
    `pnpm run build:ssr && pnpm run build:css && pnpm run generate`，和本地一致。

- **GitHub Actions workflow 文件**  
  - 放在 `.github/workflows/` 下，例如 `deploy.yml`。  
  - 内容大致包括：在 Ubuntu 上 checkout、装 Node、装 pnpm、执行上述命令、把 `dist/` 部署到 GitHub Pages（见下节示例）。

- **GitHub 仓库设置**  
  - 在仓库的 Settings → Pages 里，把「来源」设为 **GitHub Actions**（或 gh-pages 分支 / docs 目录，和 workflow 里推送的目标一致）。

### 四、触发方式（什么时候会跑一次生成）

- **按 push 触发**  
  - 例如：每次 push 到 `main` 就生成并部署。  
  - 适合：站点内容就是仓库里的 `site.json`（或由仓库里文件生成），改 JSON 即改站，push 即发布。

- **手动触发（推荐体会「点一下发布」）**  
  - 在 workflow 里加 `workflow_dispatch`，在 GitHub 的 Actions 页选该 workflow，点「Run workflow」。  
  - 不部署管理页也能实现「在网页上点一下就跑生成并发布」。

- **定时 / 其它事件**  
  - 例如 `schedule` 每天跑一次，或 `repository_dispatch` 由外部调 API 触发，适合内容来自 CMS、需要定期拉取再生成的情况。

### 五、一个最简的 workflow 示例（在做什么）

下面示例与仓库中的 [.github/workflows/deploy.yml](../.github/workflows/deploy.yml) 保持一致，便于对照上面的流程理解每一步在干什么：

```yaml
# .github/workflows/deploy.yml
name: 生成并部署静态站

on:
  push:
    branches: [main]
  workflow_dispatch:   # 支持在 GitHub 网页上手动点「Run workflow」

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: write  # 推送到 gh-pages 分支需要写权限
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: 安装 pnpm
        uses: pnpm/action-setup@v4
        # 版本由 package.json 的 packageManager 字段指定，不在此重复

      - name: 安装 Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "pnpm"

      - name: 安装依赖
        run: pnpm install --frozen-lockfile

      - name: 构建 SSR 入口
        run: pnpm run build:ssr

      - name: 构建样式
        run: pnpm run build:css

      - name: 生成静态站
        run: pnpm run generate

      - name: 部署到 GitHub Pages
        uses: peaceiris/actions-gh-pages@v4
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist
```

含义简要对应：

- **何时跑**：push 到 `main` 或手动 Run workflow。
- **权限**：`contents: write` 供 Action 推送 gh-pages 分支（否则可能报 git exit code 128）。
- **装环境**：Node + pnpm（pnpm 版本由 package.json 的 `packageManager` 指定，避免与 Action 中重复写 version 导致冲突）。
- **安装依赖**：`--frozen-lockfile` 与锁文件一致，便于复现。
- **生成**：`build:ssr` → `build:css` → `generate`，得到 `dist/`。
- **发布**：把 `dist/` 推到 gh-pages（由 `peaceiris/actions-gh-pages@v4` 完成），GitHub Pages 用这个分支更新站点。

这样就没有任何「管理页或 API 服务」需要部署，只有「仓库 + CI + GitHub Pages」。

### 六、和「部署管理页 + API」的对比

| 方面       | 不部署管理页，只用 CI 生成           | 部署管理页 + API 服务                    |
|------------|--------------------------------------|------------------------------------------|
| 运行位置   | 生成在 GitHub 的 CI 里跑             | 生成在你自己的 Node 服务器上跑           |
| 触发方式   | push / 手动 workflow / 定时等        | 打开管理页，点「生成静态站点」按钮        |
| 要维护的   | 只需维护仓库和 workflow              | 要维护服务器或 PaaS、进程、可能的环境变量 |
| 适用场景   | 内容在仓库或可由 CI 拉取；发布不频繁 | 需要随时在网页上点一下就能生成、多人共用  |

### 七、小结

「不部署管理页，只用 CI 生成」= **不在任何地方跑管理页和 API**，只在 **GitHub Actions（或其它 CI）里按和本地一样的命令生成 `dist/`，再把 `dist/` 部署到 GitHub Pages**；你要发布时，就 **push 或点一次 Run workflow**，由 CI 完成「生成 + 发布」，无需本机执行 `pnpm run dev`，也无需为管理页和 API 找服务器。

## 十三、FAQ：静态站点生成模块是否支持多用户同时在线使用

**问：将来我要将这个静态站点生成模块放到一个完整的可视化建站平台项目中，可能会有很多用户登录该平台创建并发布网站，请问这个静态网站生成模块支持多用户同时在线使用吗？**

**答：**

可以。**生成模块本身是无状态的、支持多用户并发调用**；只要集成方（平台）在调用时为每次生成提供**隔离的输入与输出**，多用户同时在线使用没有问题。当前仓库里自带的 HTTP 服务是**单租户**用法，不能直接当多用户服务用；放进完整平台时，由平台做「按请求/按站点隔离目录」或「队列/多进程」即可。

### 1. 模块层面的并发特性（支持多用户的前提）

- **无进程内共享可变状态**：`runGenerate` 是纯异步函数，入参 → 读文件 → 渲染 → 写文件 → 返回结果，没有模块级变量在多次调用间被修改。
- **输入输出完全由参数决定**：`inputPath`（读哪份网站 JSON）、`distRoot`（写到哪个目录）、`projectRoot`（只读：模板、SSR 入口、public）都是调用方传入的。不同调用只要使用**不同的 `inputPath` 和 `distRoot`**，在文件系统上就不会互相干扰。
- **渲染侧无状态**：`entry-server.js` 里每次 `renderPage` 都会 `createSSRApp` 一次，没有复用的 Vue 应用实例，请求/用户之间不会共享渲染状态。

因此：**「静态站点生成模块」在语义上支持多用户同时使用**，前提是**调用方为每次生成使用隔离的输入与输出路径**（或等价地，隔离的进程/容器）。

### 2. 当前自带的 server 为何不能多用户同时用

当前 `server.js` 里对生成 API 的调用是：`inputPath` 固定为 `projectRoot/site.json`，`distRoot` 固定为 `projectRoot/dist`。即所有请求共读一份 `site.json`、共写一个 `dist/`。若多用户同时调用 `POST /api/generate`，会读到同一份站点配置，且会把结果写进同一目录，后完成的会覆盖先完成的，无法区分是哪个用户的站点。所以：**当前这个「管理页 + 单机 Node 服务」的用法是单租户的，不能作为多用户同时在线发布**。多用户支持要靠**平台侧**在调用该模块时做隔离或排队，而不是改模块的「无状态」设计。

### 3. 集成到完整可视化建站平台时怎么做才支持多用户

要把该模块当作「多用户可同时使用」的组件，平台在调用 `runGenerate` 时需要保证**每次调用的输入与输出互不干扰**。常见几种方式：

| 方式 | 做法 | 对模块的要求 |
|------|------|----------------|
| **按请求/按站点隔离目录** | 每次「发布」为当前站点分配临时或持久目录：例如把该站点的网站 JSON 写到 `inputPath`，把 `distRoot` 设为该站点专属目录（如 `/data/builds/{siteId}/` 或临时目录）。再用同一份 `projectRoot`（共享 SSR 构建产物和模板）调用 `runGenerate`。 | 无需改模块；只需平台传入不同的 `inputPath` 和 `distRoot`。 |
| **队列串行** | 所有「生成」请求进队列，单 worker 顺序执行。每次执行时仍用该站点自己的 `inputPath`/`distRoot`，避免并发写同一目录。 | 模块不变；通过排队降低并发，避免同时写。 |
| **多进程/容器/Serverless** | 每次发布在独立进程、容器或 FaaS 中执行生成，环境天然隔离，每个环境一份自己的 `dist`（甚至自己的 `projectRoot`）。 | 模块不变；隔离由运维/部署保证。 |

只要满足「**同一时间不会有两个请求共写同一个 `distRoot`、且读的 `inputPath` 与当前站点一致**」，该模块就可以在多用户场景下安全使用。**projectRoot** 可以多请求共享（只读模板、SSR 入口、public），这样不需要为每个用户复制一整份项目，适合多租户。

### 4. 小结

- **模块层**：静态站点生成逻辑是**无状态、可并发**的，**支持多用户同时使用**，条件是由调用方为每次生成提供**独立的输入（网站 JSON）和输出目录（dist）**。
- **当前仓库的 server**：固定单份 `site.json` 和单份 `dist/`，是**单租户**用法，不能直接作为多用户并发发布。
- **放到完整可视化建站平台**：由平台在「发布」时按站点/用户传入不同的 `inputPath` 和 `distRoot`（或等价地在隔离环境中调用），即可安全地多用户同时在线使用该模块；若暂时不想处理并发，可以用队列把「生成」串行化，但模块本身并不要求必须串行。

## 十四、FAQ：升级为可视化建站平台时，静态站点生成模块需要做修改吗？

**问：如果将当前项目进一步升级为一个可视化建站平台项目，那么这个静态站点生成模块需要做修改吗？**

**答：**

**不是必须改。** 当前模块的接口（`runGenerate` 的 `inputPath`、`distRoot`、`projectRoot`）和「输入 = 完整网站 JSON、输出 = dist/」的约定已经满足平台多用户、多站点、隔离发布的需求；平台侧只要在每次发布时传入**隔离的输入与输出路径**即可直接复用。若希望平台从内存直接传网站 JSON、避免写临时文件，可以对模块做**少量可选扩展**；区块类型与渲染逻辑的扩展属于**平台集成或共享组件**的范畴，不必改模块核心。

### 1. 核心生成逻辑（runGenerate）

- **输入**：支持两种方式（二选一）：① 文件路径 `inputPath`（与现有 CLI/API 一致）；② 内存对象 `site`（已实现，见下条）。平台既可写临时文件再传 `inputPath`，也可直接传 `site`，无需再改模块。
- **输出**：`distRoot` 已由调用方指定，平台传入每站点专属目录即可。
- **运行环境**：`projectRoot` 指向「生成器」的构建产物；多用户可共享同一份 projectRoot（只读）。

### 2. 可选扩展：支持直接传入网站 JSON 对象（已实现）

- 文档 §2.2 约定模块可接受「文件路径或已读入的对象」。**当前已实现**：`runGenerate` 支持可选参数 `options.site`（完整网站 JSON 对象）。当 `options.site` 存在且为普通对象时，优先使用该对象，不再从 `inputPath` 读文件；未传 `site` 时行为与从前一致，仍从 `inputPath` 读取。因此平台可从内存直接传入网站 JSON，无需写临时文件。
- **用法示例**（平台侧）：
  ```js
  const siteJson = await getFullSiteFromDb(siteId) // 从 DB/API 拉取完整网站 JSON
  const result = await runGenerate({
    site: siteJson,
    distRoot: `/data/builds/${siteId}/dist`,
    projectRoot: generatorRoot,
  })
  ```
- 仅传 `inputPath` 的 CLI / 单机 API 无需改动，向后兼容。

**与部署脚本的关系**：若从内存传网站 JSON（`options.site`），则 `runGenerate` 只负责把传入的 site 生成到指定的 `distRoot`，**不会**执行「推送到 GitHub Pages」。因此调用方（如平台后端）需要**自己编写**「生成完成后将 dist/ 部署到 GitHub Pages」的逻辑（例如用 GitHub API 推 gh-pages、或触发 `repository_dispatch` 等），**不会走**本仓库现有的部署脚本（如 `.github/workflows/deploy.yml`）。反之，走现有部署脚本时，数据来源是仓库内的 `site.json`（由 CLI 读取），不会从内存传 site。两种方式二选一，互不交叉。

### 3. 区块与渲染层：属于平台集成，不是「改模块」

- 生成模块依赖 **SSR 入口（entry-server）→ PageRenderer → BlockRenderer** 及区块组件（如 hero、features 等）。若平台要支持**更多或不同的区块类型**，有两种常见做法：
  - **同一套组件**：平台与生成器共用同一仓库或同一 npm 包中的 BlockRenderer 与区块组件，仅扩展区块类型列表；生成器侧无需改「生成管线」，只需保证 projectRoot 指向包含这些组件的构建即可。
  - **平台自维护组件**：平台在自己的代码库中维护 entry-server + PageRenderer + BlockRenderer 及全部区块组件，构建出自身的 SSR 产物；调用时把 `projectRoot` 指向该构建目录，`runGenerate` 仍只负责「读 site JSON → 调 renderPage → 写 dist」。此时**模块核心（generate.js）仍可不改**，变的是「谁提供 projectRoot 下的 SSR 与模板」。
- 因此：**区块类型与渲染实现**由平台或共享组件包承担，不要求修改本模块的 `runGenerate` 或 I/O 约定。

### 4. 小结

| 层面 | 是否必须改 | 说明 |
|------|------------|------|
| **runGenerate 入参与 I/O** | **否** | 平台通过隔离的 inputPath / distRoot（及共享 projectRoot）即可多用户、多站点使用；无需改代码。 |
| **支持内存 site 对象** | **否（已实现）** | 已支持传入 `options.site`，平台可直接传内存中的网站 JSON，无需写临时文件。 |
| **区块/模板/SSR 入口** | **否** | 由平台或共享包提供与平台一致的组件与构建，projectRoot 指向该构建即可；不要求改生成模块核心。 |

**结论**：将当前项目升级为可视化建站平台时，**静态站点生成模块可以不做任何修改**直接接入；**传入内存 site 的扩展已实现**，平台可选用；区块与渲染的扩展在平台或共享组件侧完成即可。

## 十五、HTML 壳模板（template.html）使用与定制

### 15.1 模板在哪里使用

`src/template.html` 是静态站点生成时使用的 **HTML 壳**，在 `src/generate.js` 中被使用：

- **加载**：`getHtmlTemplate(projectRoot)` 从 `projectRoot/src/template.html` 读取模板内容。
- **填充**：对每个页面用 SSR 得到 `appHtml` 后，通过 `fillTemplate(template, replacements)` 将占位符替换为实际值，得到最终 HTML 并写入 `dist/`。

### 15.2 占位符约定

模板中**必须保留**以下占位符，生成逻辑会按约定替换：

| 占位符 | 说明 |
|--------|------|
| `{{title}}` | 页面标题（来自 page.title 或 site.name） |
| `{{baseTag}}` | `<base href="...">`（由 site.basePath 决定） |
| `{{metaDescription}}` | description 的 meta 标签 |
| `{{styleTag}}` | 指向 `assets/style.css` 的 `<link rel="stylesheet">` |
| `<!--app-html-->` | SSR 渲染出的应用 HTML（页面主体） |

修改模板时不要删除或改名上述占位符，否则生成结果会异常。

### 15.3 是否在模板中写死字体、body 样式或 Tailwind

可以在模板中增加字体链接（如 Google Fonts）、`<style>`、`<body>` 的 class（如 `bg-black text-white antialiased`）或 Tailwind CDN，**技术上可行**，且与占位符不冲突。但是否推荐取决于使用场景：

| 场景 | 建议 |
|------|------|
| **只为一个品牌/站点生成**（例如仅黄黑键） | 可以在模板里写死该站点的字体、body 样式、甚至 Tailwind CDN，与设计稿（如 `demos/yellow-keys/index.html`）保持一致。 |
| **希望生成器复用于多站点、多风格** | **不建议**在模板里写死某一套视觉。应保持模板为**最小壳**（仅含上述占位符），所有视觉（字体、背景、Tailwind 等）由**构建产物的 `style.css`** 或站点级配置控制；这样换站点只需换资源或配置，不必改模板。 |

**说明**：`<body class="bg-black text-white antialiased">` 等为 Tailwind 工具类，只有页面中已加载 Tailwind（构建出的 `style.css` 包含 Tailwind，或模板中引入 Tailwind CDN）时才会生效；否则不会报错，但无视觉效果。
