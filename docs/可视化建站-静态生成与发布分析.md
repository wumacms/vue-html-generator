# 可视化建站平台 — 静态站点生成与 GitHub Pages 发布 · 技术选型

> 平台整体设计（多用户、网站/页面/区块层级、JSON 配置化）见 [平台设计思路-多用户与JSON配置化.md](./平台设计思路-多用户与JSON配置化.md)。

## 一、场景再界定

| 环节 | 说明 |
|------|------|
| **建站方式** | 拖拽式可视化搭建；数据为 **网站 JSON / 页面 JSON / 区块 JSON**，存库并通过 API 获取 |
| **数据形态** | 区块 JSON → 区块样式；页面 JSON（含区块列表）→ 页面样式；网站 JSON（含页面列表）→ 整站 |
| **生成目标** | 根据 **网站 JSON**（含全部页面与区块）生成 **多个静态 HTML 页面**（每页一个 HTML，可带共享 CSS/JS） |
| **发布目标** | 部署到 **GitHub Pages**（纯静态托管） |

与「内容型 SSG」的区别：**内容源是「设计数据」（网站/页面/区块 JSON）而非 Markdown**，路由与页面数量由用户搭建决定，需 **JSON → Vue 渲染 → HTML** 的确定性转换。

---

## 二、数据流与架构位置

```
┌──────────────────┐      ┌─────────────────────┐      ┌────────────────────┐
│  可视化建站平台   │      │  静态站点生成器      │      │  发布               │
│  (Vue 前端)      │      │  (本仓库重点)        │      │  (GitHub Pages)     │
│                  │      │                     │      │                    │
│  拖拽 → 区块/页面 │ ───► │  网站 JSON（含页面、  │ ───► │  dist/ 推送到       │
│  多页面/多网站   │      │  区块）→ 多份 HTML   │      │  gh-pages 或 repo  │
└──────────────────┘      └─────────────────────┘      └────────────────────┘
```

生成器只关心：**输入 = 网站 JSON（含多页 + 每页区块列表）**，**输出 = 多页静态 HTML + 可选 CSS/JS/图片**。与平台内预览共用同一套「区块 JSON → BlockRenderer、页面 JSON → PageRenderer」的 Vue 组件。

## 三、技术选型结论（直接回答「哪种技术符合」）

### 推荐：**Vite + Vue 3 + 服务端渲染（SSR）做「按 schema 生成 HTML」**

| 维度 | 说明 |
|------|------|
| **为何不用 Nuxt** | Nuxt 适合「文件/目录即路由」的内容站；你的路由和页面来自 **运行时/数据库里的 schema**，不是文件系统。用 Nuxt 需要先根据 schema 动态生成一堆「虚拟页面文件」再 `generate`，复杂度高、也不自然。 |
| **为何用 Vite + Vue** | 建站平台本身大概率已是 Vue；生成侧用同一套 **组件体系**（按钮、图文、轮播等）最省事：**schema 驱动同一批 Vue 组件在 Node 里跑一遍 `renderToString`**，即可得到每页 HTML。Vite 负责打包运行时资源（若需要）和构建脚本。 |
| **为何是「SSR」** | 这里不是传统意义的「服务端渲染网站」，而是 **构建时用 Vue 的 SSR API 把「页面组件树」渲染成 HTML 字符串**，再写入文件。即：**schema → Vue 组件实例 → renderToString → 写入 dist/xxx/index.html**。 |

**一句话**：用 **Vite + Vue 3**，在 Node 构建脚本里对「每个页面 schema」调用一次 Vue 的 `renderToString`，得到多份 HTML，再配合一份 HTML 模板和（可选）共享 CSS/JS，即可生成适合 GitHub Pages 的多页静态站。

## 四、是否需要服务器端渲染？

**结论：不需要「传统意义上的」服务器端渲染。**

### 4.1 不需要：运行时 SSR（有服务器的 SSR）

- 发布目标是 **GitHub Pages**，只有静态文件，**没有 Node 服务器**。
- 不需要在用户访问时由服务器实时渲染 HTML，因此**不需要「跑着一台服务器做 SSR」**。

### 4.2 可选：构建时使用 Vue 的 SSR API

前文提到的「用 Vue SSR」指的是：**在「生成静态站」这一步**，在 Node 构建脚本里调用 `renderToString`，把 Vue 组件树转成 HTML 字符串，再写成多个 `.html` 文件。

- **本质**：是**构建时**（build time）跑一段 Node 脚本、用 Vue 的「服务端渲染 API」生成静态 HTML，**不是**用户访问时（runtime）的服务器渲染。
- **好处**：与平台预览**共用同一套 Vue 组件**（BlockRenderer / PageRenderer），一份代码两处用。
- **是否必须**：**不必须**。也可以用手写 HTML 生成器或 SPA + 静态 JSON 等方式，见下。

### 4.3 生成多页静态 HTML 的几种方式（均不需要「服务器」）

| 方式 | 是否涉及「SSR」 | 说明 |
|------|----------------|------|
| **A. 构建时用 Vue 的 `renderToString`** | 使用「SSR API」，但仅在构建时运行 | 在 Node 里跑 Vue，按页面/区块 JSON 渲染成 HTML 再写文件。与预览共用组件。 |
| **B. 手写 JSON → HTML 生成器** | 完全不涉及 SSR | 自写逻辑：区块/页面 JSON → 拼接 HTML 字符串（或简单模板）。预览用 Vue，导出用另一套逻辑，需维护两处。 |
| **C. 只发一个 SPA + 静态 JSON** | 不涉及 SSR | 一个 `index.html` + 一个 JS 包，页面与区块数据放在静态 JSON 里，由前端请求后在浏览器里用 Vue 渲染。不是「多个 HTML 页面」，而是单页 + 客户端渲染。 |

### 4.4 小结

- **需要用到服务器端渲染吗？** → **不需要**。既不需要部署 SSR 服务器，也不强制使用任何「服务端渲染」能力。
- **文档里为何会提 SSR？** → 指的是**在生成静态站时**，用 Vue 的 `renderToString` 在 Node 里跑一遍，以便复用同一套 Vue 组件；属于「构建时用 SSR API 生成静态文件」，不是「上线后做服务器端渲染」。
- **若不想碰 SSR API 可以吗？** → **可以**。采用「手写 JSON → HTML」或「单页 SPA + 静态 JSON」等方式，都无需使用 SSR。

## 五、可选方案对比

| 方案 | 做法 | 优点 | 缺点 | 与场景匹配度 |
|------|------|------|------|--------------|
| **A. Vite + Vue SSR（按 schema 渲染）** | 一个「页面渲染器」Vue 组件，接收 schema，递归渲染组件树；构建时对每页执行 `renderToString`，写 HTML 文件。 | 与建站平台组件复用、实现简单、可后续做 hydration 支持交互。 | 若输出带 Vue 运行时，体积相对大。 | ⭐⭐⭐ 最匹配 |
| **B. 纯 HTML/CSS 生成器（无 Vue 运行时）** | schema → 自写逻辑或轻量库生成 HTML 字符串 + 内联/外链 CSS，不引入 Vue。 | 输出极简、无 JS 依赖、首屏快。 | 需自己维护「组件 → HTML」的映射，和建站端组件两套逻辑。 | ⭐⭐ 适合纯展示站 |
| **C. Nuxt generate + 动态路由** | 构建前根据 schema 生成 `pages/**/*.vue` 或 `server/routes`，再 `nuxt generate`。 | 可用 Nuxt 生态。 | 要造「schema → 虚拟文件」的管线，且 Nuxt 偏重。 | ⭐ 不优先 |

**建议**：以 **方案 A** 为主实现；若后续要支持「纯静态、零 JS」的导出模式，可在 A 的基础上对同一棵组件树只取 `renderToString` 的 HTML，不注入 Vue 与业务 JS（方案 B 的子集可由 A 衍生）。

## 六、推荐技术栈与职责划分

| 层级 | 技术 | 职责 |
|------|------|------|
| **建站平台（前端）** | Vue 3 + 拖拽库（如 Vuedraggable、自定义 DnD） | 产出「项目 schema」：站点配置 + 多页 + 每页组件树（含 type、props、children、styles） |
| **静态生成（本仓库）** | Node + Vite + Vue 3（仅用 SSR 能力） | 读入 schema → 为每页创建 Vue 应用 → `renderToString` → 拼 HTML 模板 → 写出 `dist/`；可选：Vite 打一个 runtime 包供需要交互的页面引用 |
| **发布** | GitHub Actions + `peaceiris/actions-gh-pages` 或 `git push` 到 `gh-pages` 分支 / 同仓 `docs` | 将 `dist/` 部署到 GitHub Pages |

生成端**不必**使用 Nuxt；用 **Vite 仅做构建工具**（打包生成器脚本、可选客户端 JS），用 **Vue 仅做「schema → HTML」的渲染引擎** 即可。

## 七、静态生成核心流程（方案 A 细化）

1. **输入**  
   - 从建站平台 API 或导出文件获取 **网站 JSON**（与平台设计文档中的结构一致）：  
     `{ name, basePath, pages: [ { path, title, blocks: [ 区块 JSON, ... ] } ] }`。  
   - 若 API 只返回页面/区块 id，需在生成前由后端或生成脚本拉取并组装为完整网站 JSON。

2. **准备 Vue 渲染环境**  
   - 在 Node 里用 `createSSRApp` 挂载「页面渲染器」组件（`PageRenderer.vue`），内部根据页面 JSON 的 `blocks` 依次渲染 `BlockRenderer`。  
   - `BlockRenderer` 根据区块 JSON 的 `type` 映射到平台**内置区块模板**对应的 Vue 组件（与建站端预览同一套组件）。

3. **逐页生成**  
   - 对网站 JSON 的 `pages` 中每一项：  
     - 把该页的 **页面 JSON**（含 `blocks`）通过 props 传给 `PageRenderer`；  
     - 调用 `renderToString(app)` 得到该页的 HTML 片段；  
     - 将片段插入到一份 HTML 模板（含 `<!DOCTYPE html>`、`<head>`、占位符 `<!--app-html-->`、可选的 `<link>`/`<script>`）；  
     - 根据 `path` 写出文件，例如 `path: '/'` → `dist/index.html`，`path: '/about'` → `dist/about/index.html`。

4. **资源**  
   - 若需共用样式：可先打包出一份 `assets/style.css`，在模板中统一引用。  
   - 若某页需要交互：可打一份小的 runtime（例如仅含表单、轮播等），在对应页的模板中引入；多数展示页可不引 JS。

5. **输出**  
   - `dist/` 下多页 HTML + 可选 `assets/`，可直接作为 GitHub Pages 的发布目录（或通过 Actions 推送到 `gh-pages` 分支）。

## 八、GitHub Pages 相关注意点

- **仓库/组织页**：若用 `https://<user>.github.io/<repo>/`，需在生成时设 **base path**（如 `base: '/vue-html-generator/'`），所有资源引用和链接都要带该前缀；你的 schema 里可含 `basePath`，生成模板时拼进资源 URL 和 `<base href>`。
- **部署方式**：  
  - 用 GitHub Actions：在 push 或「发布」操作时运行「构建 schema → 运行生成脚本 → 输出 dist → 用 actions-gh-pages 推送到 gh-pages 或默认分支的 `docs/`」；  
  - 或本地/CI 构建后手动 `git push` 到 gh-pages。
- **多页静态**：你本来就是多 HTML，不是 SPA，所以没有 SPA 的 fallback 问题；只要 base path 正确，子路径如 `/about` 对应 `about/index.html` 即可。

## 九、总结与下一步

- **实现静态网站生成**：用 **Vite + Vue 3**，在 Node 里按 **schema 驱动 Vue 组件 + renderToString** 生成多页 HTML，最符合「拖拽建站 → 多页静态站」的场景；**不推荐**用 Nuxt 作为生成引擎。  
- **发布到 GitHub Pages**：生成结果就是普通静态文件，用现有 GitHub Actions 或 gh-pages 流程即可；注意 base path 与资源路径一致。

下一步可做的具体实现包括：  
1. 定义「项目 schema」与「单页 schema」的 JSON 结构；  
2. 实现一个最小 `PageRenderer.vue`（接收 schema、递归渲染若干原子组件）；  
3. 写 Node 脚本：读 schema → 逐页 SSR → 写 `dist/`；  
4. 在仓库中加 GitHub Actions 示例，从「构建产物」到「发布到 GitHub Pages」的完整流水线。
